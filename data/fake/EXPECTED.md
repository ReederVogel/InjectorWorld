# Fake dataset — expected import behaviour

Generated by `scripts/gen-fake-data.ts` (deterministic; PRNG seed `20260609`). Re-running
the generator reproduces these exact files. This is intentionally-FAULTY fake data for
4 launch states (CA, TX, NY, FL), used to stress-test the import pipeline + dedupe +
DataAlerts. No real PII; all names/clinics/reviews are invented.

## How to regenerate + import

```
npx tsx scripts/gen-fake-data.ts                 # writes data/fake/*.csv
npm run db:backup                                # import touches data — back up first
npm run import -- --dir ./data/fake              # imports clinics/providers/reviews
npm run scan:alerts                              # DB-wide integrity re-scan
```

Roll back any time: `npm run db:restore -- <.backups/...dump>`.

### Expected `npm run scan:alerts` result (after the import above)

```
Alerts upserted: 18   by severity: { info: 15, warning: 3 }
```

The DB scan only sees *persisted* records, so it re-detects the 3 `duplicate_clinic`
pairs (persisted) and the info-level unmatched-city / no-photo records (plus a few seed
records). It raises **no errors**, because every error-level fault above was *rejected at
import* and never persisted. Import-time alerts (the rejects) and scan alerts (persisted
imperfections) coexist in the DataAlerts collection — that is by design.

## Faults carried in the data but NOT detected today (honest gaps for Phase 4 hardening)

These rows exist so the future detectors have real-shaped dirty data, but the current
engine imports them silently (no alert):

- **Bad zip** (`ABCDE`, `00000`, `9001`) — 4 clinics. No zip validation.
- **Out-of-range latitude** (`999.123456`) — 3 clinics. Only *fully missing* coords alert;
  bad/swapped coords import silently and would mis-map.
- **Dirty phone format** (`(305) 555-0123`, `305.555.0188`, `555-0100`) — 6 providers.
  No phone normalisation.
- **Duplicate NPI alone** (same `npi_number`, different name/license) — 2 providers.
  Dedupe key is name+state+license only; NPI is not checked.
- **Brand branches** — 2 brands ("Radiance Aesthetics", "Lone Star Med Spa") × 3 branches
  each, sharing name-base + phone + website across different cities/place-ids. The
  branch auto-suggest detector does not exist yet (scan only dedupes by `google_place_id`,
  and real branches have distinct place-ids), so no alert today. Data is ready for it.

## Notes / gotchas baked into the generator

- **Review ids are offset to `rev-50000001+`** so importing this dataset never clobbers
  the ~40 seeded mock reviews (`scripts/seed-data.ts` uses `rev-00000001+`). Clinic/provider
  ids use state codes (`ca/tx/ny/fl`); seed uses city codes (`nyc/la/mia/chi/sf`) — no clash.
- **Base clinic + provider slugs are kept unique** in the generator so the *only* slug
  collision is the one curated fault (`clinic-fault-00996`). Otherwise random name reuse
  would cause unlabelled `invalid: slug` rejects and cascade their reviews.
- **Literal duplicate-id rows** (`clinic-ny-00001`, `prov-ny-00088`, first review) appear
  twice to test upsert idempotency → counted as `updated`, no alert.
- CSV is RFC-quoted: any field with a comma/quote/newline (directions_url lat,lng;
  apple_maps_url; hours_json; provider responses) is double-quoted with `""` escaping.
- Cities that match a seeded metro Location (so clinics surface on city pages): CA = Los
  Angeles / San Francisco / San Diego; TX = Houston / Dallas / Austin; NY = New York;
  FL = Miami.
